# 要件定義書 (Requirements Definition)

## プロジェクト概要

ユーザーの「マイページ」を作成するプロジェクト。
ポートフォリオ、サーバー情報、活動記録などを集約したWebサイトを構築する。
**本プロジェクトはGitHubのPublicリポジトリとして公開される。**

## 技術スタック (Tech Stack) - 詳細版

本プロジェクトでは、**「自宅サーバー運用における安全性」と「開発・デプロイの効率化」**を両立するため、以下のスタックを選定する。

### 1. アプリケーション層

* **フロントエンド**: Astro + React
  * **選定理由**: 静的コンテンツ（ブログ、ポートフォリオ）の高速表示と、動的コンテンツ（3Dビューワー、サーバー監視）の柔軟なコンポーネント管理を両立するため。
* **バックエンド (BFF/監視API)**: TypeScript (Hono または Fastify)
  * **選定理由**: フロントエンドと型定義を共有可能であり、かつ軽量・高速に動作するため。Proxmox APIとの通信およびデータの整形（サニタイズ）を担当する。
* **メディア処理**:
  * **画像**: `Astro Image` コンポーネントを使用し、WebP形式へ自動変換・最適化を行う。
  * **3Dモデル**: `.glb` 形式を採用し、Draco圧縮を適用する。Lazy Loadを実装し、初期ロード時間を短縮する。

### 2. インフラ・コンテナ基盤

* **コンテナランタイム**: Docker / Docker Compose
  * **選定理由**: 環境差異を排除し、移植性と復旧容易性を高めるため。
* **Webサーバー / リバースプロキシ**: Nginx
  * **選定理由**:
    * **セキュリティ**: アプリケーションサーバーを直接インターネットに露出させず、SSL終端やヘッダー制御を行うため。
    * **パフォーマンス**: 静的アセットのキャッシュ制御やロードバランシングを行うため。

### 3. CI/CD (継続的インテグレーション/デプロイ)

* **CIツール**: GitHub Actions
  * **役割**: コードのLint、Test、ビルド、およびDockerイメージの作成とGHCR (GitHub Container Registry) へのPush。
* **CDツール**: Watchtower
  * **役割**: 自宅サーバー上で動作し、定期的にレジストリを監視してコンテナを自動更新する。
  * **選定理由 (Pull型デプロイ)**: 外部からのWebhookを受け取るためのポート開放が不要であり、自宅サーバーのファイアウォール設定を堅牢に保ったまま自動デプロイを実現できるため。
* **公開・ネットワーク基盤**: Cloudflare Tunnel
  * **役割**: 自宅サーバーの特定ポート（Nginxコンテナ）をCloudflareのエッジネットワークへ安全にトンネリングする。ポート開放や固定IPを不要にする。

## 機能要件

### 1. コンテンツ (Content)

* **自己紹介 (About Me)**
  * **名前**: ハンドルネーム / 本名
  * **出身地**
  * **所属大学**
  * **所属サークル**
  * **趣味**
  * **所有資格**
  * **技術スタック**: 扱える言語・ツール等
  * **3D制作**: 使用ソフトやスキルセット（技術スタックの一部としても可）
* **計算機構成 (Computer Configuration)**
  * **物理マシン構成 (Physical Hosts)**:
    * ノートPC, デスクトップPC, サーバー等のスペック記述 (CPU, メモリ, ストレージ等)。
    * タイトルは「サーバー」と強調せず、計算機全体として記述する。
  * **ネットワーク構成 (Network Diagram)**:
    * 仮想ルーター, 仮想スイッチ, 物理スイッチ等の配線図・構成図。

    ### サーバーについて

    * **リアルタイムサーバーリソースモニター**:
      * CPU使用率、メモリ使用率、ディスク使用率、ネットワーク使用率等を表示。
      * webhook, webRTCなどを利用してリアルタイム同期を目指す。
    * **サービス一覧 (Services)**:
      * 稼働中のソフトウェア一覧をテーブル形式で表示。
      * 項目: 名前, 簡単な説明, 役割。
      * タグ付け機能とソート機能を実装 (各VMごとに表示可)。
    * **運用記 (Operation Log)**:
      * サーバー運用に関する簡易的なログ記録 (ブログより軽量なもの)。
      * **フォーマット**: Markdown形式。
      * **画像管理**: 画像ファイルはリポジトリ内の `public/assets` に配置する。
      * **執筆ワークフロー**:
        1. ローカル等でMarkdownを作成・配置。
        2. Pull Requestを作成。
        3. `main` ブランチへマージされるとGitHub Actionsが走り、自動デプロイされる。
    * **過去の構成 (History)** (Future Feature):
      * 過去の構成へ遡って閲覧できるページ (優先度低)。
    * **将来の構想 (Future Plans)**:
      * 仮想化プラットフォーム, PBS (Proxmox Backup Server), Docker管理, クラスター移行への予定等。
* **制作物 (Works / Services)**
  * 制作したサービス・プロダクト一覧。
  * **ギャラリー (Gallery)**:
    * 撮影した写真を掲載するページ。
    * グリッドレイアウト等で見やすく表示する。

* **活動記録 (Activities)**
  * 大学・サークルでの活動履歴。
    * **3Dモデルビューワー**:
      * ファイル形式: `.glb` (glTF Binary) 標準 + Draco圧縮。
      * **UXフロー (Lazy Load)**:
        1. ページ読み込み時は「プレビュー画像」のみを表示し、重い3Dモデルは読み込まない。
        2. ユーザーが画像をクリックしたタイミングで、専用のビューワーページへ遷移またはモーダル展開し、そこで初めて3Dモデルをロード・表示する。
      * 挙動: 自動回転 + ユーザー操作（視点移動）。
  * **ブログアクセスパネル**:
    * Qiita, Note, Zenn の記事集約 (RSS/API利用)。
* **リンク集 (Links)**
  * SNS (Twitter/X, etc.), GitHub, Email。
* **アクセス解析 (Analytics)**:
  * **ツール**: Cloudflare Web Analytics (プライバシー重視)。
  * **義務**: サイト内にプライバシーポリシーを記述し、解析ツールの利用を明示する。

### 3. 機能要件 - 詳細版 (Functional Requirements - Detailed)

#### 1. サーバーリソースモニター (Secure Real-time Monitor)

自宅サーバーのリソース状況を可視化する機能。セキュリティを最優先事項とし、BFF (Backend For Frontend) アーキテクチャを採用する。

* **システム構成**:
  * **フロントエンド (Astro)**: 直接Proxmoxと通信せず、自作の「監視用API」に対してリクエストを行う。
  * **監視用API (Middleware)**:
    * Proxmox APIトークンを保持し、内部ネットワーク経由でProxmoxへアクセスする。
    * 取得した生データ（Raw Data）から、IPアドレス、MACアドレス、内部IDなどの機密情報を除去（サニタイズ）する。
    * フロントエンドには、表示に必要な最小限のデータ（CPU使用率、メモリ使用率、稼働状態など）のみをJSON形式で返却する。
* **導入理由**:
  * **機密情報の隠蔽**: クライアントサイド（ブラウザ）にAPIトークンやサーバーの内部構成情報を一切露出させないため。
  * **攻撃対象領域の縮小**: 万が一Webサーバーが侵害されても、Proxmox本体への管理権限（root等）は奪われない設計とする（APIトークンには PVEAuditor 権限のみを付与）。

#### 2. ネーミングコンベンション (Naming Convention)

サーバー運用者の「遊び心」と「管理のしやすさ」を表現するコンテンツ。

* **概要**: 運用している仮想マシン(VM)やホスト名の命名規則を紹介するセクション。
* **規則**: **「日本の軍艦・護衛艦の名前」**を採用。
* **表示内容**:
  * 各VMの名前（例: Kaga, Izumo など）とその由来。
  * その艦（サーバー）が担っている役割（例: 旗艦＝メインルーター、工作艦＝ビルドサーバーなど）の解説。

### 2. デザイン・UI/UX

* **テーマ切り替え機能**
  * 通常モード (Standard): 黒ベース。
  * ターミナル風モード (Terminal Mode): CUIライクな見た目。
* **ページ構成**
  * Astro ファイルベースルーティング (SPA風遷移検討)。

## 非機能要件

* **公開設定**: GitHub Public Repository
  * APIキーや個人情報（非公開にすべきもの）の管理に注意する（`.env`利用など）。
* **CI/CD (Continuous Integration / Continuous Deployment)**:
  * **GitHub Actions**を積極的に活用し、自動化を行う。
  * **チェック項目**: `build`, `lint` (ESLint), `type-check` (TypeScript), `format-check` (Prettier) 等。
  * **自動デプロイ**: `main`ブランチにマージされた時点でビルド等の処理が走り、即座に本番環境へ反映される構成にする。
* **開発プロセス**:
  * **Pull Request (PR) 運用**: 変更は必ずPRを通して行い、直接pushは避ける。
  * **承認制**: PRのマージには承認(Review/Approve)を必須とする。
* **その他**:
  * **ブラウザ互換性**: 主要なモダンブラウザ (Chrome, Firefox, Safari, Edge) での動作を保証する。
  * **レスポンシブ対応**:
    * 必須要件。ただし、PC画面のデザイン・実装を優先する。
    * モバイルファーストでなくとも良いが、スマホでも閲覧可能にする。
  * **パフォーマンス**: Lighthouseスコアで **90点以上** を目標とする。
  * **SEO対策**: 検索エンジン最適化を意識したマークアップを行う。
  * **依存関係管理**: **Dependabot** を設定し、自動で依存関係の更新PRを作成するようにする。

### 非機能要件 - 詳細版 (Non-Functional Requirements - Detailed)

#### 1. 可用性とデプロイ運用 (Availability & Operations)

* **ゼロダウンタイム・デプロイメントの追求**:
  * Watchtowerの設定において、既存コンテナを停止する前に新しいイメージのダウンロード（Pull）を完了させる設定を適用する。
  * Nginxを前段に配置し、バックエンド更新中の一時的なエラーハンドリングや、将来的なBlue/Greenデプロイへの移行を可能にする構成とする。
* **自動化の範囲**:
  * GitHubの main ブランチへのマージをトリガーとして、ビルドから本番環境への反映までを無人で完結させる。

#### 2. セキュリティ (Security)

* **ネットワーク分離**:
  * Web公開用のコンテナ群（Nginx, Astro, API）は、自宅ネットワーク内のDMZ（または専用VLAN）に配置し、内部の重要セグメント（管理LAN）への通信をファイアウォールで厳格に遮断する。
  * **デプロイ先**: 自宅サーバー上に構築したWeb公開専用のVM。
    * 構成: 1つのVM内に Nginx（リバースプロキシ）コンテナ、Astro（フロントエンド）コンテナ、APIコンテナを同居させる（Docker Compose構成）。
    * 外部公開: Cloudflare Tunnel (`cloudflared`) コンテナを同居、またはホスト側で実行し、Nginxコンテナへトラフィックを流す。
* **最小権限の原則 (Principle of Least Privilege)**:
  * 監視用APIが使用するProxmoxのAPIトークンには、監視に必要な「読み取り専用権限」のみを付与し、操作権限（再起動、作成、削除等）は持たせない。
